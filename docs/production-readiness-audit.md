# 生产级电商秒杀项目审计与重构路线图

## 1. 目标确认

本项目目标不是做“看起来像生产”的 Demo，而是按真实可落地生产系统的标准设计和实现一套商城与高并发秒杀系统。项目暂不实际部署上线，但代码、架构、文档、测试、风险分析和运行材料都应达到可以被工程团队评审、开源发布和技术复盘的标准。

目标形态：

- C 端商城：商品浏览、搜索、详情、购物车、下单、支付、订单、秒杀频道。
- C 端秒杀：活动预热、场次切换、预约提醒、开抢倒计时、抢购排队、结果轮询、支付倒计时、售罄与降级提示。
- B 端运营：商品管理、活动管理、场次管理、秒杀商品报名/审核、库存配置、活动数据看板、风控配置。
- 高并发能力：限流、削峰、缓存、幂等、防超卖、防重复下单、异步建单、消息可靠投递、补偿任务、监控告警。
- 工程交付：Docker Compose、本地一键运行、API 文档、架构图、流程图、压测脚本、风险分析、README、项目说明。

## 2. 当前项目现状

### 2.1 已具备的基础

当前仓库已经具备微服务雏形：

| 模块 | 当前职责 |
| --- | --- |
| `mall-gateway` | API 网关、JWT 校验、路由、TraceId 透传 |
| `mall-auth` | 登录、刷新 Token |
| `mall-user` | 注册、当前用户、内部用户凭证查询 |
| `mall-product` | 类目、商品、SKU 查询，商品详情 Redis 缓存 |
| `mall-cart` | 购物车增删改查、购物车结算 |
| `mall-order` | 普通订单、取消、支付状态更新、基础秒杀流程 |
| `mall-inventory` | 库存锁定、释放、扣减 |
| `mall-payment` | 模拟支付、本地消息、支付事件发布 |
| `mall-web` | Vue 3 C 端商城页面草稿 |
| `deploy` | MySQL、Redis、Nacos、RocketMQ、RocketMQ Dashboard |

已经存在的可保留能力：

- Flyway 管理数据库迁移。
- MyBatis Plus 基础 CRUD。
- Redis 商品缓存、空值缓存、TTL 抖动、简单重建锁。
- 库存 `available_stock` / `locked_stock` 模型。
- 普通订单支付后扣减库存。
- 秒杀 Redis Lua 预扣库存和用户 SKU 幂等键。
- RocketMQ 事件雏形。
- 本地消息表雏形。
- 网关 JWT 鉴权和 TraceId。
- 前端已经有 C 端商城视觉基础。

### 2.2 当前生产级结论

当前项目是一个不错的“微服务商城 baseline”，但还不是生产级商城系统。主要差距不在于是否能运行，而在于：

- 业务域不完整：缺少活动、促销、秒杀场次、预约、订单超时、售后、运营后台。
- 高并发链路不完整：缺少活动预热、资格令牌、动态限流、热点隔离、降级、排队、压测验证。
- 一致性机制不完整：本地消息没有补偿扫描任务，跨服务事务存在悬挂和不一致风险。
- 安全治理不足：硬编码 JWT secret、开放 CORS、无 RBAC、内部接口无服务间鉴权。
- 工程质量不足：没有测试、没有 CI、没有压测脚本、没有风险演练。
- 前端存在演示兜底数据，必须改为 profile/mock 隔离，不能混在生产 API 层。

## 3. 关键风险审计

### 3.1 P0 风险：上线前必须解决

| 风险 | 当前表现 | 影响 | 改造方向 |
| --- | --- | --- | --- |
| 秒杀库存初始化接口暴露给普通登录用户 | `SeckillController` 的 `/orders/seckill/stocks` 没有管理员权限控制 | 任意用户可修改秒杀库存 | 拆到 B 端 promotion/admin 服务，增加 RBAC 和操作审计 |
| 秒杀没有活动模型 | 只有 `skuId + stock`，无活动、场次、开始/结束时间、限购、状态 | 无法支撑淘宝/京东式秒杀 | 新增 promotion/seckill 域和活动表 |
| 秒杀 Lua 只校验库存和用户 SKU | 不校验活动状态、场次、用户资格、限购、令牌、风控 | 高并发下容易被脚本直接刷接口 | 引入秒杀令牌、限流、风控、活动状态缓存 |
| 本地消息无补偿任务 | 只 `saveAndSend`，失败后落表但没有定时重试 | MQ 短暂不可用会导致业务事件丢失 | 增加 outbox publisher、重试、死信、告警 |
| 跨服务调用参与本地事务 | 订单创建事务中调用库存锁定；支付事务中调用订单标记已支付 | 可能出现远端成功、本地回滚的不一致 | 使用 Saga/Outbox/补偿任务，减少事务内远程调用 |
| 普通下单无业务幂等键 | `OrderService.create` 每次请求都生成新订单 | 网络重试可能重复下单、重复锁库存 | 引入 `requestId/clientOrderToken` 唯一约束 |
| 订单没有超时取消 | 待支付订单不会自动释放库存 | 库存长期锁死 | 增加延迟消息或定时任务释放库存 |
| 内部服务可被绕过网关访问 | 服务端口直接暴露，本地服务只信任 header | 伪造用户 header 可访问内部接口 | 内部网络隔离、服务间签名、mTLS 或内网鉴权 |
| JWT secret 硬编码 | 多个 `application.yml` 中有 demo secret | 密钥泄露风险 | 使用环境变量、密钥管理、启动校验 |
| CORS 允许任意来源携带凭证 | `addAllowedOriginPattern("*")` + `allowCredentials(true)` | 浏览器侧安全风险 | 按环境配置白名单 |

### 3.2 P1 风险：生产可用性增强

| 风险 | 当前表现 | 改造方向 |
| --- | --- | --- |
| 没有限流规则落地 | 依赖 Sentinel 依赖和 dashboard 配置，但无规则初始化 | 网关、秒杀、登录、下单接口配置限流和熔断 |
| MQ 消费幂等不足 | 订单支付事件依赖订单状态判断，缺少消费日志 | 增加消费幂等表和消息业务唯一键 |
| 库存锁表约束不足 | `wms_stock_lock` 缺少 `order_no + sku_id` 唯一约束 | 加唯一键、库存流水、锁定批次 |
| 商品缓存更新策略不足 | 只有读缓存，缺少 B 端变更后的失效/重建策略 | 商品变更事件驱动缓存失效 |
| 支付链路过度简化 | 只有模拟成功，无支付中、失败、关闭、回调验签 | 增加支付状态机和回调幂等 |
| 数据库缺少审计字段体系 | 有基础时间字段，但缺少操作人、版本号、删除态 | 补充统一审计、乐观锁、逻辑删除策略 |
| 前端 mock 混入真实 API | `mall-web/src/api/mall.ts` 失败后自动原型兜底 | 改为 `VITE_USE_MOCK` 显式开关 |
| 缺少配置中心治理 | Nacos 只做发现，配置仍在本地 yml | 生产 profile 接入配置中心或环境变量规范 |

### 3.3 P2 风险：工程展示和长期维护

| 风险 | 当前表现 | 改造方向 |
| --- | --- | --- |
| 没有测试 | 未发现单元测试、集成测试、前端测试 | 补 JUnit、Testcontainers、Vitest/Playwright |
| 没有压测 | 无 JMeter/k6/Gatling 脚本 | 增加秒杀压测脚本和结果报告 |
| 没有 CI | GitHub 发布后无法自动构建验证 | 增加 GitHub Actions |
| 没有架构图 | README 只有文本说明 | 增加系统架构图、核心流程图、时序图 |
| Docker Compose 只含中间件 | Java 服务未容器化 | 增加服务镜像构建和本地 compose profile |

## 5. 秒杀目标业务流程

### 5.1 活动预热

1. B 端创建秒杀活动、场次、活动商品、活动库存、限购规则。
2. 审核通过后，活动进入预热状态。
3. 预热任务将活动商品、场次、库存、限购规则加载到 Redis。
4. C 端读取活动页静态化/缓存数据，避免直接打 DB。
5. 开抢前生成或开放秒杀资格令牌接口。

### 5.2 C 端抢购

1. 用户进入秒杀频道，选择场次。
2. 未开始时展示倒计时和预约提醒。
3. 开抢后按钮变为“立即抢购”。
4. 前端获取秒杀令牌或挑战校验。
5. 提交秒杀请求：`activityId + sessionId + skuId + requestId + token`。
6. 网关限流，风险服务检查用户/设备/IP。
7. Redis Lua 原子校验：
   - 活动是否开始；
   - 商品是否属于活动；
   - 令牌是否有效；
   - 用户是否已买/超过限购；
   - 活动库存是否充足；
   - 请求是否重复。
8. 预扣成功后写入结果 `ACCEPTED`，投递 RocketMQ。
9. C 端进入“排队中”，轮询结果。

### 5.3 异步建单

1. 消费秒杀消息。
2. 通过 `requestId` 幂等校验。
3. 创建秒杀订单，订单状态为待支付。
4. 写订单项、订单快照、活动信息快照。
5. 记录库存流水或库存锁。
6. 写结果 `CREATED + orderNo`。
7. 发送订单超时关闭延迟消息。

### 5.4 支付与库存最终确认

1. 用户在支付倒计时内支付。
2. 支付服务生成支付单并处理回调。
3. 支付成功事件驱动订单变为已支付。
4. 库存从活动预扣转为最终扣减。
5. 支付超时或取消订单，释放库存和用户购买标记。

## 9. 工程重构路线图

### Phase 1：生产级基础治理

目标：把当前 baseline 从 Demo 运行态提升为可靠工程基础。

交付：

- 移除默认 API 自动 mock 兜底，改为显式 `mock` profile。
- JWT secret、数据库密码、Redis 密码全部环境变量化。
- CORS 按环境白名单。
- 增加 B 端/内部接口权限模型。
- 增加 Feign 超时、重试边界和错误码治理。
- 增加基础单元测试和集成测试框架。
- 增加 GitHub Actions 构建。

验收：

- `mvn test` 和 `npm run build` 通过。
- 无硬编码生产密钥。
- 普通用户不能访问运营/库存初始化接口。

### Phase 2：交易一致性改造

目标：解决普通订单、库存、支付之间的生产级一致性。

交付：

- 普通下单幂等键。
- 订单超时关闭和库存释放。
- 库存锁唯一约束和库存流水。
- 支付状态机、支付回调幂等。
- 本地消息 outbox 扫描重试。
- MQ 消费幂等表。

验收：

- 重复提交不会重复下单。
- 支付重复回调不会重复扣库存。
- MQ 短暂失败后可自动补偿。
- 订单超时后库存释放。

### Phase 3：生产级秒杀域

目标：从 SKU 库存接口升级为完整秒杀活动系统。

交付：

- `mall-promotion` 或 promotion 包。
- 活动、场次、活动 SKU、预约、限购。
- 活动预热到 Redis。
- 秒杀令牌。
- Redis Lua 校验活动、令牌、库存、限购、幂等。
- MQ 异步建单。
- 秒杀结果查询。
- 失败补偿和库存回滚。

验收：

- 未开始不能抢。
- 已结束不能抢。
- 售罄不会超卖。
- 同一用户不能重复抢同一活动商品。
- 高并发压测下 Redis 库存和订单数一致。

### Phase 4：C 端和 B 端成熟原型

目标：让项目在 GitHub 和技术评审中体现完整产品能力。

交付：

- 重做 C 端秒杀频道和详情页。
- 增加订单列表、支付倒计时、抢购结果体验。
- 增加 B 端运营后台页面。
- 前后端接口对齐，不再依赖隐式 mock。

验收：

- C 端能完整体验普通购物和秒杀购物。
- B 端能配置活动并影响 C 端展示。
- 页面状态与真实后端状态一致。

### Phase 5：可观测性、压测和风险分析

目标：补齐企业级上线前流程。

交付：

- Micrometer 指标。
- Prometheus/Grafana Compose profile。
- JSON 日志和 traceId 贯穿。
- k6/Gatling 压测脚本。
- 秒杀压测报告。
- 风险清单、降级预案、故障演练文档。

验收：

- 能看到 QPS、RT、错误率、Redis/MQ/DB 指标。
- 秒杀压测输出可复现。
- README 能说明瓶颈和扩容策略。

### Phase 6：GitHub 发布材料

目标：让项目具备完整的开源发布材料。

交付：

- 完整 README。
- 架构图、流程图、ER 图。
- 本地运行说明。

## 10. 企业级风险分析框架

后续每个阶段完成后，都按以下风险维度复查：

| 维度 | 检查问题 |
| --- | --- |
| 容量风险 | 峰值 QPS、Redis/DB/MQ 承载量、扩容方式是否明确 |
| 一致性风险 | 远程调用失败、MQ 失败、重复消息、事务回滚是否有补偿 |
| 安全风险 | 鉴权、权限、接口暴露、密钥、CORS、内部调用是否安全 |
| 风控风险 | 刷接口、脚本、重复抢、黑产账号、IP/设备限频是否覆盖 |
| 可用性风险 | 限流、熔断、降级、排队、只读缓存是否可用 |
| 数据风险 | 库存、订单、支付、消息是否可追溯、可修复 |
| 运维风险 | 日志、指标、告警、健康检查、启动顺序是否清楚 |
| 发布风险 | 配置隔离、数据库迁移、回滚方案是否明确 |
| 测试风险 | 单测、集成测试、并发测试、压测是否覆盖核心链路 |

## 11. 下一步执行建议

建议按 Phase 1 开始，不先做更多页面。原因是当前生产级最大短板是基础治理和交易一致性。如果继续堆页面，会放大后续返工。

下一步具体任务：

1. 将前端原型兜底数据改为显式 mock profile。
2. 增加管理员/运营权限模型，封禁普通用户访问库存初始化和 B 端接口。
3. 增加普通下单幂等键和订单超时字段。
4. 增加库存锁唯一约束和库存流水。
5. 增加本地消息重试任务。
6. 补核心链路测试。

完成这些后，再进入秒杀活动域和 C/B 端产品原型重构。
