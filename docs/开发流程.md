一、项目启动与需求定义阶段
核心目标：对齐业务价值，明确需求边界，评估项目可行性与投入产出。
参与角色：业务方、产品经理、项目经理、技术架构师、财务 / 合规负责人
关键动作
需求调研与梳理
产品经理通过业务访谈、用户调研、竞品分析，挖掘核心业务诉求，区分「必须实现的核心需求」与「可后置的次要需求」，梳理完整业务流程与规则。
需求文档化
输出产品需求文档（PRD），包含业务背景、功能清单、交互原型、业务规则、验收标准、异常场景处理等，作为后续所有环节的基准依据。
可行性评估
技术团队评估技术可行性、技术栈选型、研发周期与人力投入；同时完成合规评估（数据安全、行业监管要求）、成本评估与风险预判。
立项与评审
输出项目立项报告，明确项目目标、里程碑、资源预算、风险应对方案，通过公司级 / 部门级立项评审后，项目正式启动。
需求变更管控
建立正式的需求变更流程：所有需求调整必须提交变更申请，评估对进度、成本、架构的影响，经审批通过后方可纳入开发范围，避免需求蔓延导致项目失控。
核心产出
PRD 文档、需求规格说明书、产品原型、立项报告、需求跟踪矩阵
二、架构与方案设计阶段
核心目标：确定技术实现路径，统一设计规范，前置化解技术风险。
参与角色：系统架构师、技术负责人、开发组长、测试负责人、安全工程师、运维负责人
关键动作
整体架构设计
从业务架构、技术架构、数据架构三个维度完成顶层设计：
技术架构：确定技术栈、中间件选型、服务拆分策略、部署拓扑、高可用 / 高并发方案
数据架构：数据库选型、分库分表策略、数据流转、缓存方案、数据备份与容灾
详细设计落地
拆解到模块级别，输出模块划分、接口定义（请求 / 响应格式、错误码规范）、数据库表结构、核心业务逻辑流程图、异常处理机制。
跨角色设计评审
组织开发、测试、运维、安全、业务多方参与评审，排查方案漏洞、性能隐患、安全风险与运维痛点，确认风险应对方案，评审通过后方可进入开发。
开发规范对齐
统一代码规范、接口规范、数据库设计规范、日志与埋点规范、错误码规范，保障团队协作一致性。
核心产出
架构设计文档、详细设计说明书、数据库设计文档、API 接口文档、技术规范手册
三、开发与协同管控阶段
核心目标：高效、高质量完成代码开发，通过工程化手段保障协作效率与代码质量。
参与角色：前后端开发工程师、测试工程师、DevOps 工程师
关键动作
任务拆解与迭代规划
项目经理 / 开发组长将需求拆解为可落地的开发任务，对应到迭代周期（敏捷模式通常为 2 周一个迭代），明确每个任务的交付时间与验收标准。
环境与工程化准备
DevOps 团队统一搭建开发环境、联调环境，配置代码仓库、分支管理规则与 CI 流水线，统一依赖管理与构建工具。
代码开发与自测
开发人员遵循分支管理规范（如 GitFlow、主干开发模式）在独立分支开发；同步编写单元测试用例，完成模块级功能自测，保障基础逻辑正确性。
代码评审（Code Review）
代码合并到主干前，必须经过同事 / 技术组长评审，重点检查代码规范、逻辑正确性、性能隐患、安全漏洞与可维护性，评审不通过不得合并。
持续集成（CI）
代码提交后自动触发流水线：执行代码编译、单元测试、静态代码扫描、安全漏洞扫描、代码规范检查，任一环节不通过则阻断合并，从源头管控代码质量。
核心产出
功能代码、单元测试用例、代码评审记录、联调通过的功能模块
四、测试与质量保障阶段
核心目标：多维度验证系统质量，提前发现并修复缺陷，保障上线后稳定性。
参与角色：测试工程师、开发工程师、产品经理、安全工程师
关键动作
测试策划与用例设计
测试团队基于 PRD 与设计文档编写测试计划与测试用例，覆盖功能场景、边界场景、异常场景与业务主流程。
多轮分层测试
功能测试：验证需求功能是否符合预期，发现缺陷后提交 Bug 跟踪系统，驱动开发修复
集成测试：验证模块间、上下游系统间的联调逻辑与数据流转是否正常
系统测试：在完整测试环境中做端到端全流程验证，同步开展兼容性测试、易用性测试
专项测试：针对企业级系统的核心要求开展专项验证，包括性能压测、安全渗透测试、稳定性测试、容灾切换测试
回归测试
Bug 修复后、版本转测前，全面验证缺陷修复效果，同时确认未对原有功能造成影响。
用户验收测试（UAT）
在预发布环境中，由业务方 / 产品经理按照真实业务场景做最终验收，确认功能符合业务预期，验收通过后方可申请上线。
核心产出
测试用例、测试报告、缺陷跟踪报告、UAT 验收报告、性能 / 安全专项报告
五、发布与部署上线阶段
核心目标：低风险、可回滚地完成版本发布，保障生产环境平滑过渡。
参与角色：DevOps / 运维工程师、开发工程师、测试工程师、项目经理、业务负责人
关键动作
上线审批与方案准备
提交上线申请，经过技术、业务、安全、运维多方审批；同步制定详细上线方案，包含发布步骤、回滚方案、应急预案、值班人员安排。
预发布验证
在与生产环境配置完全一致的预发布环境中，执行最终的功能与流程验证，排除环境差异导致的问题。
灰度 / 分批发布
企业级系统普遍采用低风险发布策略：
金丝雀发布：先切少量流量到新版本，观察无异常后逐步扩大流量
蓝绿发布：保留两套环境，新版本验证通过后一键切换流量
分批发布：按机器批次逐步升级，每批次完成后巡检确认
上线后巡检
发布完成后，实时监控系统指标（CPU、内存、接口响应时间、错误率）、业务指标（订单量、用户量）与日志异常，持续观察 1~2 小时无异常后，发布流程才算完成。
核心产出
上线方案、发布执行记录、上线巡检报告、回滚记录（如有）
